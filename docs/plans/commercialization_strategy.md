# OmniSaver 商业化与合规策略草案

## 核心模式：本地计算 + 移动端协同 (Local-First + Mobile Companion)

### 1. 架构分离，风险隔离
*   **后端引擎 (OmniSaver Core)**: 以 Docker 镜像形式**免费/开源分发**，用户自行部署在 NAS、软路由或个人服务器上。
    *   **合规优势**：所有抓取行为（IP）、账号风险（Cookie/Session）、数据存储（本地硬盘）均由用户本人承担。你仅提供“工具”，不触碰、不存储任何第三方平台的受保护内容。这属于典型的“避风港”或技术中立原则。
*   **移动端 App (OmniSaver Companion)**: 提供 iOS/Android 应用，主要功能是“链接接收与转发器”和“内容阅读器”。
    *   **商业化路径**：App 可以采用**付费下载（买断制）**或**轻量级订阅（解锁高级同步/AI功能）**。
    *   **核心功能**：用户在抖音/Instagram/推特点击“分享”，选择发送给 OmniSaver App，App 再将链接通过用户的内网穿透或自定义 API 发送给家里的 NAS 进行下载。

### 2. 为什么这种模式风险极低？
1.  **没有中心化的 SaaS 服务器**：你不需要维护庞大的服务器集群去抗平台的反爬虫（这既烧钱又极易被起诉）。如果平台封锁 IP，封的是用户家里的宽带 IP。
2.  **不构成“分发”与“盗版”**：用户抓取的内容永远锁在他们自己的物理设备里（Localhost/NAS），没有任何一个环节经过你的服务器进行再分发。
3.  **技术上极难取证**：平台方（如微信、小红书）无法证明是你作为一个公司在系统性地扒取他们的数据，他们看到的只是一堆分散的家用宽带在零星访问。

### 3. App 的核心卖点与增值服务 (如何赚钱？)
既然核心抓取引擎免费，App 必须提供极强的便利性和附加价值：
*   **无缝转发 (Seamless Forwarding)**：解决用户在外用手机看到好内容，却无法方便地扔给家里 NAS 的痛点（替代你现在的 Telegram Bot 方案，因为很多人不用 TG）。
*   **离线缓存与阅读 (Offline Reading)**：App 可以自动从用户的 NAS 同步已下载的文本和缩略图，提供极佳的原生阅读体验（类似 Readwise/Cubox），并在 Wi-Fi 下缓存视频。
*   **云端桥接服务 (Cloud Relay - 增值订阅)**：如果用户没有公网 IP，你可以提供一个小型的中转服务器（只转发 URL，不传文件数据），让用户在外也能轻松触发 NAS 下载。
*   **AI 智能标签与总结 (AI Summary - 增值订阅)**：在 App 端集成你自己的 Gemini/OpenAI 接口，帮用户自动总结每天归档的文章和长视频。

### 4. 法律免责与合规设计
在 App 的启动页和官网购买页必须明确：
*   **"OmniSaver 是一款个人数字资产管理工具，不提供任何内容托管服务。"**
*   **"用户需自行提供运行环境并对下载内容的合法性与版权负责，禁止用于任何商业分发。"**

### 5. 总结
**这套“Docker 免费核心 + App 收费客户端”的模式是目前个人知识管理工具（PKM）领域最安全、最被验证的商业化路径。** 它巧妙地将法律风险完全转移给了终端用户，同时又通过极其便利的移动端体验和高级的 AI 处理功能建立了自己的商业护城河。
